# Scaling Options

## 1. Vertical scaling
- Increase CPU/RAM on a single server.
- Works early-stage, limited by single host resources.

## 2. Horizontal scaling
- Add more server instances behind a load balancer.
- Stateful connections require sticky-session or shared state.

## 3. Redis adapter (Socket.IO)
- `socket.io-redis` or `@socket.io/redis-adapter`.
- Pub/sub channel sync events across instances.
- Example:
```js
const io = new Server(server);
const { createAdapter } = require('@socket.io/redis-adapter');
const { createClient } = require('redis');

const pubClient = createClient({ url: 'redis://localhost:6379' });
const subClient = pubClient.duplicate();

io.adapter(createAdapter(pubClient, subClient));
```

## 4. Sticky session / session affinity
- Required on typical HTTP load-balancers to keep a client on same backend.
- Can use `source-IP` or `cookie` affinity.

## 5. Message brokers
- Use Apache Kafka/RabbitMQ for non-realtime state sync and events.
- Good for analytical dashboards and persisted history patches.

## 6. Serverless WebSocket gateways
- AWS API Gateway WebSocket, Azure Web PubSub, etc.
- Handles scaling and connection management.

## 7. Partition by room / shard
- Route certain game rooms to same backend to reduce cross-traffic.
- Use namespace/room key hashing.
